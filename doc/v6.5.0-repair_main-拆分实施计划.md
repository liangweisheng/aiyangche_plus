### 1.3 共享模块设计

#### `cloudfunctions/<func>/common/auth.js`

```javascript
// 驻留在每个云函数目录的 common/ 子目录中
// 被 repair_main 和 repair_aux 各自持有同步副本
// ⚠️ 修改时需同步两处：repair_main/common/auth.js ↔ repair_aux/common/auth.js
//
// 提供：
//   - createAuthModule(db, _, permissions) → { checkPermission, _checkShopAccess, _getCallerRecord, _validatePhoneAccess }
//
// 依赖：db, _ (通过工厂函数参数注入，避免顶层 require 顺序问题)
```

#### `cloudfunctions/<func>/common/db-utils.js`

```javascript
// 驻留在每个云函数目录的 common/ 子目录中
// 被 repair_main 和 repair_aux 各自持有同步副本
//
// 提供：
//   - fetchAllOrders(db, _, where, field)
//   - MAX_LIMIT = 100
//
// 依赖：db, _ 通过函数参数注入
```

**为什么不用 `require('../common/...')`？**

微信云开发部署云函数时，IDE 只打包该云函数目录内的文件。`require('../common/auth')` 指向父目录，父目录不会被上传，云端运行时会报 `Cannot find module` 错误。

**替代方案**：`common/` 作为云函数内部子目录，`require('./common/auth')` 在部署时随云函数一同打包上传。两个云函数各自持有一份副本，需保持同步。`common/` 中的代码极少改动（~300行），维护成本可控。

### 7.2 部分回滚（只回退某阶段）

| 失败阶段 | 回滚命令 | 影响范围 |
|----------|----------|----------|
| Phase A 失败 | `git reset --hard HEAD~1` | 无影响 |
| Phase B 失败 | `rmdir /s /q cloudfunctions\repair_aux` + `git checkout -- cloudfunctions/repair_main/` | 无影响 |
| Phase C 失败 | `git reset --hard HEAD~1`（回到 SPLIT-B） | 无影响 |
| Phase D 失败 | `git checkout -- utils/util.js` | 无影响 |
| Phase E 部署失败 | 重新部署 `repair_main`（覆盖部署） | 短暂不可用 |

### 7.3 回滚安全原则

1. **绝不手动改已提交的代码**，只使用 git reset/checkout
2. **部署回滚时先部署 repair_main（恢复旧版），后删除 repair_aux**
3. **回滚后必须跑 `npx jest --verbose` 确认 154 测试通过**

## ⚠️ 共享模块同步维护规范

`repair_main/common/` 和 `repair_aux/common/` 中的文件是**两份同步副本**。修改任一处时需同步到另一处。

### 同步原则

| 操作 | 做法 |
|------|------|
| 修改 auth.js 或 db-utils.js | 修改一份后，手动复制到另一份 |
| 新增 shared function | 两个 common/ 目录都加 |
| 删除 shared function | 两个 common/ 目录都删 |

### 快速同步命令（Windows）

```bash
rem 从 repair_main 同步到 repair_aux
copy /Y cloudfunctions\repair_main\common\auth.js cloudfunctions\repair_aux\common\auth.js
copy /Y cloudfunctions\repair_main\common\db-utils.js cloudfunctions\repair_aux\common\db-utils.js
```
