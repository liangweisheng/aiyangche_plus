# AI养车门店管理系统 v5.0.0 开发方案 — AI月报功能

> **版本**：v5.0.0（规划中）
> **规划日期**：2026-05-01
> **基于版本**：v4.6.0（向后兼容）
> **功能定位**：每月1日自动生成上月经营诊断报告，基于规则引擎给出健康评分 + 可执行建议 + 标杆案例对标

---

## 一、决策总表

| 维度 | 决策 |
|------|------|
| **AI 能力** | v5.0 规则引擎 → v5.x 预留大模型接口 |
| **数据来源** | 自店历史数据 + 内置行业基准值 + 优秀案例图 |
| **客单价基准** | 自动计算本店均值 + 允许手动调整 |
| **目标客群** | 小微型汽修门店（1-2 工位）|
| **展示入口** | Dashboard 摘要卡片 → 点击跳转独立 `monthlyReport` 详情页 |
| **核心模块** | 经营概览 / 健康评分雷达图 / 诊断建议 / 行业对比 |
| **分析维度** | 新客占比 / 维保占比 / 增值占比 / 老客复购率 / 客单价趋势 |
| **触发策略** | 首次使用弹窗引导填(工位数+开业年份)→ 之后每月自动生成 |
| **案例展示** | 点击指标卡片弹出对应标杆案例图（动态匹配）|
| **计算时机** | 云函数定时器每月1日凌晨自动计算 → 存入数据库 |
| **历史月报** | 支持近3个月切换查看 |
| **门店信息收集** | 工位数 + 开业年份（首次弹窗 + 设置页可改）|

## 四、规则引擎设计

### 评分体系（满分100分，5个维度各20分）

| 维度 | 权重 | 数据来源 | 计算逻辑 |
|------|------|---------|---------|
| **新客获取** | 20分 | `首次进店车牌 / 当月总进店车牌` | ≥30%=20分, 20-30%=15分, 10-20%=10分, <10%=5分 |
| **维保业务** | 20分 | `(保养+维修类工单) / 总工单` | ≥70%=20分, 50-70%=15分, 30-50%=10分, <30%=5分 |
| **增值渗透** | 20分 | `含增值项目的工单 / 总工单` | ≥20%=20分, 10-20%=15分, 5-10%=10分, <5%=5分 |
| **客户粘性** | 20分 | `有历史记录的车牌 / 当月总进店车牌` | ≥70%=20分, 50-70%=15分, 30-50%=10分, <30%=5分 |
| **客单价水平** | 20分 | `当月平均客单价 / 行业基准` | ≥基准120%=20分, 80-120%=15分, 50-80%=10分, <50%=5分 |

### 小微型门店（1-2工位）基准值（初版）

| 指标 | 优秀 | 良好 | 一般 | 偏低 |
|------|------|------|------|------|
| 月工单量 | ≥60单 | 40-60单 | 20-40单 | <20单 |
| 平均客单价 | ≥¥400 | ¥250-400 | ¥150-250 | <¥150 |
| 新客占比 | ≥30% | 20-30% | 10-20% | <10% |
| 维保占比 | ≥70% | 50-70% | 30-50% | <30% |
| 增值项目占比 | ≥20% | 10-20% | 5-10% | <5% |
| 老客复购率 | ≥70% | 50-70% | 30-50% | <30% |

### `repair_main` 新增 action

```
名称: monthlyReportGenerator

动作: 调用 generateMonthlyReport 为每个活跃门店生成上月报告
条件: 仅处理上月在 repair_orders 中有记录的门店
```

## 七、开发里程碑

### Phase 1: 基础框架 + 数据管道

**目标**：搭好骨架，数据能跑通

- [ ] **1.1 新建 monthlyReport 页面骨架**
  - 创建 `pages/monthlyReport/` 目录及4个文件（js/wxml/wxss/json）
  - 基础页面结构：顶部导航栏（含月份切换器）、内容区域（各模块容器）
  - 页面路由注册到 `app.json`
  - 月份切换器 UI：支持近3个月切换（如 [3月] [4月] 当前选中高亮）

- [ ] **1.2 云函数 generateMonthlyReport / getMonthlyReport**
  - 在 `repair_main/index.js` 中新增 `generateMonthlyReport` action
  - 核心逻辑：
    1. 入参接收 `{ yearMonth: "2026-04" }`
    2. 按 shopPhone + 时间范围查询 repair_orders（上月1号~月末）
    3. 聚合计算基础指标（营收、工单数、客单价、不重复车牌数）
    4. 计算5项核心指标（新客/维保/增值/复购/客单价趋势）
    5. 调用规则引擎计算健康评分和诊断结果
    6. 写入/更新 `repair_monthlyReports` 集合
    7. 返回完整的报告数据
  - 新增 `getMonthlyReport` action：按 shopPhone + yearMonth 查询并返回已有报告
  - 新增 `listRecentReports` action：返回最近N个月的报告摘要列表

- [ ] **1.3 定时触发器配置**
  - 配置云函数定时触发器 `monthlyReportGenerator`
  - cron 表达式：每月1日 00:05 CST（`0 5 1 * *`）
  - 触发逻辑：遍历所有上月有订单的活跃门店，逐个调用 generateMonthlyReport
  - 幂等处理：已存在该月报告的跳过不重复生成

- [ ] **1.4 门店信息引导弹窗 + 设置页入口**
  - 新建 `components/shop-guide/` 组件（3个文件）
  - 弹窗内容：
    - 标题："完善门店信息，开启经营诊断"
    - 表单项：工位数（数字输入，默认2）、开业年份（年份选择器，默认当年）
    - 底部按钮：「开始使用」+「稍后再说」
  - 逻辑：
    - 首次进入 dashboard 时检查 `shopProfile` 是否存在
    - 不存在则弹出引导弹窗
    - 用户填写后调用 `updateShopProfile` 云函数保存
    - 选择「稍后」设置标记，下次不再自动弹出（可在设置页手动填写）
  - 在 `pages/mine/mine.wxml` 和 `mine.js` 中新增「门店设置」入口卡片
  - 设置项：工位数、开业年份、城市（选填）、客单价基准修正（选填）

- [ ] **1.5 数据库集合准备**
  - `repair_monthlyReports` 集合：无需预建（云数据库无 Schema），确保权限规则正确
  - 权限：仅创建者（云函数）可读写，客户端仅读

### Phase 2: 规则引擎 + 评分算法

**目标**：纯逻辑层，与UI解耦，可独立测试**

- [ ] **2.1 monthlyReportEngine.js 核心逻辑**
  - 创建 `utils/monthlyReportEngine.js`
  - 主函数 `generateReport(rawData, shopProfile)` 接收原始聚合数据 + 门店配置
  - 输出完整报告对象（与数据模型一致）
  - 纯函数设计：无副作用，无网络请求，输入→输出确定性强

- [ ] **2.2 5项指标计算函数**
  - `calcNewCustomerRatio(orders, allHistoricalPlates)` — 新客占比
    - 当月首次出现车牌 / 当月总不重复车牌
  - `calcMaintenanceRatio(orders)` — 维保占比
    - 从 serviceItems 关键词判断工单类型（保养/维修 vs 非维保）
  - `calcValueAddedRatio(orders)` — 增值项目占比
    - 使用 VALUE_ADDED_KEYWORDS 匹配 serviceItems
  - `calcRepeatRate(monthlyPlates, historicalPlates)` — 老客复购率
    - 有历史记录的车牌 / 当月总进店车牌
  - `calcAvgTicket Trend(currentAvg, prevAvg)` — 客单价趋势
    - 返回 `{ trend: 'up'|'down'|'stable', changePercent: number }`

- [ ] **2.3 健康评分算法**
  - `calcHealthScore(metrics, benchmarks)` 函数
  - 5个维度各20分，查基准值表映射到分数
  - 输出 `{ total, level, dimensions }`
  - level 分级：excellent(≥85), good(70-84), warning(50-69), critical(<50)

- [ ] **2.4 诊断规则匹配引擎**
  - `matchDiagnoses(metrics, healthScore)` 函数
  - 遍历 DIAGNOSIS_RULES，condition 为真则生成一条诊断
  - 最多返回3-4条（优先 warning > info > success）
  - 模板变量替换（{value} → 实际数值）
  - 每条诊断关联 caseTag 用于后续匹配案例图片

- [ ] **2.5 benchmarks.js 基准值配置**
  - 创建 `utils/benchmarks.js`
  - 按门店规模（小微型1-2工位）导出基准值对象
  - 包含所有阈值常量
  - 预留扩展接口：后续可加中型/大型基准值

- [ ] **2.6 diagnosisRules.js 诊断规则配置**
  - 创建 `utils/diagnosisRules.js`
  - 导出 DIAGNOSIS_RULES 数组 + VALUE_ADDED_KEYWORDS 数组
  - 规则与引擎分离，方便后续调整规则而不改代码逻辑

  - **无数据/新店**：开店不到1个月 → 显示「积累更多数据后即可生成报告，通常需要满1个月经营数据」

  - **定时器失败**：某门店生成失败 → 下次打开月报页时检查，若无报告则实时触发生成（降级策略）

## 附录B：术语说明

| 术语 | 说明 |
|------|------|
| yearMonth | 报告月份，格式 "YYYY-MM"，如 "2026-04" 表示2026年4月的报告（数据来自4月1日~4月30日）|
| bayCount | 门店施工工位数量，用于对标不同规模门店的基准值 |
| openYear | 门店开业年份，用于判断新老店（新店阈值放宽）|
| healthScore | 健康综合评分，满分100，由5个维度各20分加权 |
| caseTag | 诊断规则关联的案例标签，用于匹配对应的标杆案例图片 |
| VALUE_ADDED_KEYWORDS | 增值项目关键词列表，用于从 serviceItems 中识别增值服务 |
| monthlyReportGenerator | 云函数定时触发器名称，每月1日凌晨自动生成上月报告 |
