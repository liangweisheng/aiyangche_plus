## 六、开发方法论沉淀（重要！）

从「员工手机号 bug 反复修改十余次」等经验中提炼：

### ❌ 错误做法（耗时 4 小时，5 次失败）
```javascript
// 猜测式提问 → 方案导向修改
Q: "用 shopPhone 行不行？"     → ❌ 还是店主手机号
Q: "用 cloudRecord.phone？"   → ❌ 也是店主记录
Q: "加多重条件判断兜底？"      → ❌ 条件复杂且未验证
```

### ✅ 正确做法（耗时 1.5 小时，1 次成功）
```javascript
// 验证式提问 → 数据驱动修改
Q: "先加日志看实际值是什么？"
→ 日志揭示：welcome.js 已正确写入员工手机号到 shopInfo.phone
→ 根因：proUnlock 页面读取后被云端店主记录覆盖
→ 方案：用 addedBy 判断是否员工 → 直接用缓存的 phone（1行解决）
```

### 方法论公式
```
遇到 Bug 时：
  ① 在关键节点加 console.log（入参/中间变量/返回值）
  ② 真机复现，运行一次，收集日志
  ③ 根据日志精确定位失败节点（"原来这里的值已经是错的了"）
  ④ 基于真实数据设计最简方案（1行能解决就不用3行）
  ⑤ 清理调试日志（或降级为 warn）

禁止行为：
  ✗ 凭记忆/直觉猜测字段含义
  ✗ 未验证实际值就提出"换个字段试试"
  ✗ 用更复杂的条件掩盖根因
```

## 十、关键经验与教训汇总

### 经验 1：调试方法论（ID: 43015945 / ID: 94558585）
> **先定位再动手，加日志定位根因**

### 经验 2：多端适配（ID: 59691365 / ID: 26858283）
> Donut 多端模式下 wx.cloud.init 必须传入 appid 和 envid

### 经验 3：跨账号架构（ID: 77330490）
> 使用 new wx.cloud.Cloud({ resourceAppid, resourceEnv }) 访问资源方

### 经验 4：Pro 状态判断（ID: 27140884）
> code 有值即已激活 + expireTime 未过期（不能用 code === unlockKey 判断）

### 经验 5：车牌选择器参数（ID: 79793069）
> max-height=60vh，最近车辆 limit=3，键盘上限系数 0.40

### 经验 6：报表缓存策略（多次记录）
> 本周/本月/本年每日仅拉取一次，今日 tab 实时拉取

### 经验 7：v4.0.0 开发红线（ID: 33244367）
> 绝对不能影响已发布版本的正常使用（云函数/数据库/app.js 向后兼容）

### 经验 8：提问优先级原则
> 1. 第一优先级："当前的实际数据值是什么？"
> 2. 第二优先级："数据在哪个环节发生了变化？"
> 3. 第三优先级："这个方案的假设前提是否成立？"
> 4. 禁止：未看到真实数据就提出"换个字段/换个方案"
