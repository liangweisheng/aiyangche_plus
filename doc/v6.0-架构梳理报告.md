## 二、核心架构问题诊断

### 问题 1：repair_main 单文件 32 个 action — 严重过度膨胀 ⏸️ 暂不实施

**现状**：2159 行代码，32 个 action 挤在一个 `index.js` 中。

**暂不实施原因**（2026-05-09 确认）：
- 虽然文件大，但采用路由模式（`switch(action)` 分发），各 action 彼此独立，改一个不影响另一个
- 拆分成本高：32 个 `callFunction` 调用点全部要改路由，任何遗漏即线上故障
- 拆分后公共函数（`fetchAllOrders`、`checkPermission` 等）需重复或抽包，引入新复杂度
- 部署从 1 个变 6 个，调试 `console.log` 分散
- 当前运行正常，拆分风险大于收益

**如果未来某个 action 独立需要扩容/限流/独立部署时再拆分**。

### 问题 2：客户端 DB 查询与云函数查询混用 — 数据获取策略不一致 ✅ 已解决

**现状（Phase 4 前）**：同一页面中，部分数据走客户端 DB，部分走云函数，缺乏统一策略。

| 数据获取方式 | 使用页面（Phase 4 前） |
|-------------|---------|
| 纯云函数 | dashboard, report, monthlyReport（3 个） |
| 纯客户端 DB | carSearch, carAdd, carDetail, orderList, orderDetail, memberAdd, memberList, checkSheet, checkSheetList, checkSheetDetail, proUnlock（11 个） |
| 混合 | welcome, carList, orderAdd, dataExport（4 个） |

**现状（Phase 4 后）**：

| 数据获取方式 | 使用页面 |
|-------------|---------|
| 纯云函数 | dashboard, report, monthlyReport, carList, orderList, memberList, checkSheetList, dataExport（8 个）✅ |
| 纯客户端 DB | carSearch, carAdd, carDetail, orderDetail, memberAdd, checkSheetDetail, proUnlock（7 个，均无分页需求） |
| 混合 | welcome, orderAdd, checkSheet（3 个） |

**已解决**：
- ✅ carList/orderList/memberList/checkSheetList/dataExport 迁移到云函数查询
- ✅ 客户端 DB 残留均为无分页需求的单条/少量查询，延迟敏感场景（如 carSearch 搜索）
- ✅ 消除了客户端 `.limit(20)` 数据丢失风险
- ✅ 云函数端统一鉴权，数据安全性提升

### 问题 3：权限检查散落且不统一 ✅ 已解决

**现状（Phase 2 前）**：

| 权限维度 | 检查位置 | 一致性 |
|---------|---------|--------|
| isStaff | carDetail, orderDetail, dataExport, custom-tab-bar | ⚠️ 部分页面遗漏 |
| isSuperAdmin | dataExport, proUnlock | ⚠️ 仅2处 |
| isPro | report, orderAdd, memberAdd, dataExport, 云函数addStaff | ⚠️ 前后端都有，不统一 |
| isOwner | 仅 proUnlock 内部变量 | ⚠️ 非全局方法 |
| isGuest | dashboard, memberList, report, proUnlock | ⚠️ 各自判断逻辑不同 |

**已解决**：
- ✅ 云函数端 32 个 action 全部配置权限等级（`ACTION_PERMISSIONS`）
- ✅ 云函数端统一鉴权函数 `checkPermission()` + `_getCallerRecord()` + `_validatePhoneAccess()`
- ✅ 前端 `app.js` 新增 `checkPageAccess()` 守卫 + `isGuest()` / `isStaff()` / `isSuperAdmin()` / `isRegistered()` 全局方法
- ✅ 7 个页面接入 `checkPageAccess`：carAdd('registered')、carList('admin')、checkSheet('registered')、checkSheetList('registered')、memberList('admin')、monthlyReport('admin+pro')、dataExport('superAdmin+pro')
- ✅ 删除旧 `_validateWriteAccess`，统一由 `checkPermission` 承担

**剩余说明**：carDetail / orderDetail / orderAdd / memberAdd 等页面仍有手写权限判断（如 `isStaff()` 控制按钮），属于页面内部 UI 逻辑（"进来了能看什么"），与 `checkPageAccess`（"能不能进"）是不同层面，不冲突，无需统一。

### 问题 4：数据模型设计不合理 — repair_activationCodes 成为"万能表" ⏸️ 暂不实施

**现状**：`repair_activationCodes` 同时存储：
- 店主记录（type='free', role='admin'）
- 员工记录（type='staff', role='staff'/'admin'）
- 激活码信息（code, unlockKey, expireTime）
- 门店配置（bayCount, openYear, city, name, shopTel, shopAddr）
- 门店码（shopCode）
- 导航配置

**暂不实施原因**（2026-05-09 确认）：
- 查询都走 `shopPhone` 索引，性能没有瓶颈
- 拆分需要数据迁移（线上真实数据），不可逆，任何一个环节出错即数据丢失
- 客户端所有 `db.collection('repair_activationCodes')` 调用都要改，改动面大
- 当前运行正常，数据迁移风险极高

**如果未来单条记录字段超过微信云开发上限或查询性能出现瓶颈时再拆分**。

### 问题 5：缓存策略各页面各自实现 — 无统一基础设施 ⏸️ 暂不实施

**现状**：

| 页面 | 缓存方式 | TTL | 一致性保证 |
|------|---------|-----|-----------|
| carList | 双层 Storage 缓存 | base=5min, agg=2min | 静默刷新 |
| report | 按日 Storage 缓存 | 当天有效 | 强制刷新可清除 |
| proUnlock | 多个 Storage key | 永久 | 无TTL |
| orderAdd | servicePhrases Storage | 永久 | 无 |
| dashboard | 内存缓存 | 页面内有效 | 无 |

**暂不实施原因**（2026-05-09 确认）：
- v6.0 Phase 4 已将 5 个列表页迁到云函数，客户端缓存需求趋势是**减少**的
- 三种缓存模式（TTL/按日/永久）语义本质不同，抽象后调用者仍需自己选模式，认知负担未减少
- 迁移有缓存格式不兼容风险，升级后首次访问变慢
- 运行正常，收益太小，不动

**如果未来出现以下情况可重新评估**：
- 新增 3+ 个页面需要缓存且模式可统一
- Storage 接近 10MB 上限需要缓存淘汰策略
- 需要跨页面缓存协调

### 问题 6：多端兼容逻辑侵入核心流程 ⏸️ 暂不实施

**现状**：`app.js` 中多端相关代码约 200+ 行，散布在：
- `_isMultiEndMode()` 检测
- `_autoLoginMultiEnd()` 登录
- `_validateMultiEndCache()` 验证
- `_enterGuestMode()` 多端跳转
- `onLaunch()` 分支
- `_restoreShopInfo()` 缓存来源标记

**问题**：
- 核心 `onLaunch` 流程中 if/else 分支增加，阅读困难
- 缓存隔离（`_platform` 标记）增加了维护负担

### 4.3 权限中间件层 ✅ 已完成

**方案**：在云函数端增加统一的鉴权中间件，每个 action 声明所需权限：

```javascript
// 云函数权限配置
var ACTION_PERMISSIONS = {
  // 认证域 - 无需鉴权
  getOpenId: 'public',
  loginByPhoneCode: 'public',
  registerShop: 'public',

  // 门店域 - 管理员
  updateShopInfo: 'admin',
  updateShopProfile: 'superAdmin',
  getShopProfile: 'registered',

  // 核心业务 - 注册用户
  addCar: 'registered',        // ★ 补充鉴权
  createOrder: 'registered',
  editOrder: 'registered',
  voidOrder: 'admin',          // 仅管理员可作废

  // 员工域 - 超级管理员+Pro
  addStaff: 'superAdmin+pro',
  removeStaff: 'superAdmin',
  updateStaffRole: 'superAdmin',
  listStaffs: 'admin',

  // 统计域 - 管理员
  getDashboardStats: 'registered',
  getReportOrders: 'admin',
  getCustomerRanking: 'admin',
}

// 统一鉴权函数
async function checkPermission(action, event) {
  var required = ACTION_PERMISSIONS[action]
  if (required === 'public') return true
  if (required === 'registered' && !event.shopPhone) return false
  if (required === 'admin' && !(await _isAdmin(event))) return false
  if (required === 'superAdmin' && !(await _isSuperAdmin(event))) return false
  if (required.includes('+pro') && !await _isPro(event)) return false
  return true
}
```

```javascript
    if (!data || !data._date) return false
    return data._date === util.formatDate(new Date())
  }
}
```
